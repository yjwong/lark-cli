# General parameters

To meet diverse application scenarios, Lark Open Platform provides a rich set of APIs and events for developers to call or subscribe to. These APIs and events involve some general parameters, which are introduced here in a unified manner.

General parameters can be used as request parameters or response parameters. Before using the APIs and events provided by the Lark Open Platform, it is recommended to first understand the meanings of these general parameters.

## App-related parameters

### app_id

`app_id` is the unique identifier of Lark Open Platform app. It is automatically generated by the system when the app is created and cannot be modified by users. You can check the `app_id` in the **Credentials & Basic info** page of the [Developer Console](https://open.larksuite.com/app?lang=en-US).
![](//sf16-sg.larksuitecdn.com/obj/open-platform-opendoc-sg/1d42a4316b6c1a2c42fb7479aa2b6dd6_19vDwdfybs.png?height=501&lazyload=true&width=1152)

### app_secret

`app_secret` is the secret key of the app. It is automatically generated by the system when creating an app and is required as one of the request parameters when calling certain APIs. You can check or reset it on the **Credentials & Basic Info** page of the [Developer Console](https://open.larksuite.com/app?lang=en-US).
![](//sf16-sg.larksuitecdn.com/obj/open-platform-opendoc-sg/12a240b743df4b996c2127a8b6efaacd_JmuDttb5oA.png?height=498&lazyload=true&width=1121)

### app_ticket

`app_ticket` is a field used by store apps to improve security.
When you call the API to get `app_access_token` of the store app, you need to include `app_ticket` as one of the request parameters.
After creating a store app and configuring the request address of the event, Lark Open Platform will push `app_ticket` to the app once per hour, and users can also call the API to actively trigger the push of `app_ticket`.

## Organization-related parameters

### tenant_key

`tenant_key` is a tenant unique identifier, which corresponds to an enterprise  in actual application scenarios. Apps can obtain the `tenant_key` in the following ways: 
- When the organization installs the app, Lark Open Platform pushes it to the app through events.
- When the user logs in and grant authorization, the Lark Open Platform will return the `tenant_key`.

### deparment_id

A custom department ID, or if there is no custom department ID, Lark Open Platform will generate a random string as the custom department ID.

> **Note:** Since April 2021, the Open Platform has updated the interface policy for department management, allowing the reuse of deleted department_id. As a result, the uniqueness of department_id is no longer guaranteed. It is not recommended to rely on it as a unique identifier for departments in apps. `open_department_id` is recommended to be used instead.

### open_department_id

The open_department_id is an automatically generated department ID by the Open Platform, starting with `od-`.  Each `open_department_id` is globally unique, ensuring uniqueness across apps and development subjects.

## User-related parameters
  
Lark employs a well-designed user identity system, providing user identifiers such as `user_id`, `open_id`, and `union_id` to meet various development scenarios. For detailed information, see ID Concepts.
  
When developing an app, consider the following logical selection of IDs:
- If data association between different apps is not required, `open_id` can be used.
- If data association between apps is needed, and the developers belong to the same organization, `union_id` can be used.
- If data associationbetween apps is needed, and the apps may be developed by different organizations, but the users of the apps belong to the same organization, `user_id` is the appropriate choice.

## Access tokens related parameters

To enhance API call security, Lark Open Platform has implemented an access token mechanism. Before invoking an API, an app must obtain the necessary access credentials, also call `access_token`.

`access_token` represents the authorization granted to an app by the platform, tenant (organization or team), or user. There are three types of access tokens:
- App access token, with the parameter name `app_access_token`.
- Tenant access token, with the parameter name `tenant_access_token`.
- User access token, with the parameter name `user_access_token`.

To learn how to get access tokens, see Get access token.

For a comprehensive understanding of the Lark Open Platform's permission system, see Authentication and authorization.

### app_access_token

The `app_access_token` is the access token for apps. It identifies the app's identity to the Open Platform, allowing the app to access its own related information without specifying an organization or user. 

For instructions on how to obtain the app access token, see Get app_access_token for store apps and Get app_access_token for custom apps.

### tenant_access_token

The `tenant_access_token` is the access token for a tenant. With the `tenant_access_token`, an app can perform operations on behalf of an organization or team, such as retrieving information about a contact user. 

To learn how to get `tenant_access_token`, see Get tenant_access_token for store apps, Get tenant_access_token for custom apps.

### user_access_token

The `user_access_token` is the access token for a user. By using the `user_access_token`, an app can perform operations on behalf of a user, such as creating a cloud document or schedule through the API. 

To learn how to get `user_access_token`, see Getting user_access_token.

## Other parameters

### chat_id

`chat_id` is a unique identifier for a conversation, including both private and group chats.

### open_message_id

`open_message_id` is a unique identifier for the message body.
