# Event Overview

Through event subscription, applications can respond to change events in Lark in a timely manner. When an event occurs, the open platform will send event messages according to the subscription method you configured. See event list for the supported events.

## Use Cases

Here are two simple use cases to illustrate the scenarios where developers can subscribe to different events based on their actual needs.

- **Real-time data processing**
   
   If an app requires real-time synchronization of data changes from Lark, it can subscribe to events. For example, when a user leaves the company, the app needs to process the business data of the departing user immediately. In this case, the app can subscribe to the user deletion event.

- **Quick event response**
  
  If an app needs to respond promptly to user actions, it can subscribe to events. For instance, when a new member joins a group, the group bot may want to send a welcome message to the new member. To achieve this, the app can subscribe to the user added to group event.

## Subscription process

The process for event subscription is as follows:

![](//sf16-sg.larksuitecdn.com/obj/open-platform-opendoc-sg/df5a33e6df5991202606decab5e90d51_vDUVdXEReo.png?height=168&lazyload=true&maxWidth=750&width=2714)

1. Optional: Configure Encryption Key. To ensure data security, optionally configure an encryption key. This will enable encrypted event push to the specified subscription method.

2. Optional: Modify Verification Token. Upon app creation, an automatic verification token is generated by the Open Platform. Users have the option to modify this token. Including the token in the pushed events allows the app to verify their validity.

3. Configure subscription method. Configure a subscription method to receive event messages pushed by the open platform to the application. When the event subscribed by the application occurs, the open platform will send the event message in the specified manner.

4. Add Events. Specify the events that the app needs to monitor and receive notifications for.

5. Apply for Permissions. Obtain the required permissions for the subscribed events to take effect. Apply for permissions, create an app version, and submit it for review. Once the review is passed, the event subscription will become active.

6. Receive and Process Events. When an event occurs, the Open Platform will push the event to the app. The app needs to receive and process the event accordingly.

## Event versions

Open Platform provides two versions of events: v1.0 and v2.0. We recommend you to subscribe to the v2.0 events, as they offer more comprehensive and reasonable settings.

To avoid receiving duplicate events, ensure that you don't subscribe to the same event multiple times. For example, if you subscribe to both `Contact Changes v1.0` and `Employee Deletion v2.0` events, your app will receive two identical employee deletion events when an employee leaves.

## Event Structures

Different event versions have different event structures. When adding an event, it can be determined from the **Add Event** page which version of the event is being added.

### Event structure of v1.0

Here is an example of a v1.0 event:
- The `ts` field represents the time when the event was sent, which is generally close to the time of the event.
- The `uuid` field is the unique identifier of the event.
- The `token` field is the verification token.
- The `event` structure contains detailed information about the event. The information varies depending on the event type, which can be determined by the `event.type` field.

```json
{
    "ts": "1502199207.7171419",
    "uuid": "bc447199585340d1f3728d26b1c0297a",
    "token": "41a9425ea7df4536a7623e38fa321bae",
    "type": "event_callback",
    "event": {
        "app_id": "cli_9c8609450f78d102",
        "chat_id": "oc_26b66a5eb603162b849f91bcd8815b20",
        "operator": {
            "open_id": "ou_2d2c0399b53d06fd195bb393cd1e38f2",
            "user_id": "gfa21d92"
        },
        "tenant_key": "736588c9260f175c",
        "type": "p2p_chat_create",
        "user": {
            "name": "user_name",
            "open_id": "ou_7dede290d6a27698b969a7fd70ca53da",
            "user_id": "gfa21d92"
        }
    }
}
```

### Event structure of v2.0

Here is an example of a v2.0 event:
- The `schema` field represents the version of the event. This field is added in the v2.0 event structure to differentiate it from the v1.0 event structure.
- The `trace_id` field is the unique identifier of the event.
- The `app_id` field represents the ID of the app.
- The `event` field contains detailed information about the event. The event type can be determined by the `event.type` field.

```json
{
    "schema": "2.0",
    "header": { 
        "event_id": "f7984f25108f8137722bb63cee927e66",
        "token": "066zT6pS4QCbgj5Do145GfDbbagCHGgF",
        "create_time": "1603977298000000",
        "event_type": "contact.user_group.created_v3",
        "tenant_key": "xxxxxxx",
        "app_id": "cli_xxxxxxxx",
    },
    "event":{
    }
}
```

## Event notification

### Notification frequency

When subscribed events occur, Lark will send event data in JSON format via an HTTP POST request to the preconfigured subscription method.

Upon receiving the HTTP POST request, the app needs to respond within 3 seconds with an HTTP 200 status code. Otherwise, Open Platform considers the push failed and will make up to 4 retries with intervals of 15 seconds, 5 minutes, 1 hour, and 6 hours.

Based on the above description, the maximum time window for event retransmission is approximately 7.1 hours. Handle and eliminate duplicate events within this time window. Use the following methods to determine event uniqueness:

- For version 1.0 events, use the `uuid` field in the event structure.
- For version 2.0 events, use the `event_id` field in the event structure.

### Event notification order

To ensure the availability of user events and maintain consistency between internal and external data changes, Open Platform uses **ordered events** for certain event notifications. This means that the next event is pushed only after the previous event has been successfully received by the user.

For ordered events, users need to ensure the proper consumption of corresponding preceding and following events to avoid event blockage or untimely reception. Specifically, when an ordered event becomes blocked, subsequent events will directly enter the retry queue and wait in line. Only after the preceding event is successfully processed will the retry attempts of the queued events at the front begin, starting with the first retry attempt.

To determine if an event is ordered, refer to the detailed description provided in the corresponding event document.
